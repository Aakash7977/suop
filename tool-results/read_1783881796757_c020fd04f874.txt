    17→## 2. Business Purpose
    18→
    19→### Why This Screen Exists
    20→The RBAC module is the **security backbone** of the ERP. Every action across all 200+ modules is gated by permissions defined and managed here. Without this module functioning correctly, the entire ERP security model collapses.
    21→
    22→### Who Uses It
    23→| Persona | Department | Usage |
    24→|---|---|---|
    25→| System Administrator | IT | User provisioning, role assignment, account lock/unlock |
    26→| Security Officer | Compliance | Permission audits, role reviews, delegation management |
    27→| HR Manager | HR | User onboarding via invitations, designation updates |
    28→| Department Head | Operations | Approval authority configuration, delegation setup |
    29→| Auditor | Internal Audit | Login history review, session audit, permission matrix review |
    30→
    31→### Business Problems Solved
    32→1. **User Provisioning**: Create user accounts via invitation flow
    33→2. **Access Control**: Assign roles with specific permission sets
    34→3. **Security Monitoring**: Track login attempts, sessions, device history
    35→4. **Account Security**: Lock/unlock accounts, revoke sessions, force password changes
    36→5. **Delegation**: Temporarily delegate approval authority during absence
    37→6. **Compliance**: Audit trail for every user/role/permission change
    38→
    39→### Downstream Dependencies
    40→Every module in the ERP depends on RBAC:
    41→- **Procurement**: `po:approve`, `po:create`, `po:read`
    42→- **Inventory**: `inventory:post`, `inventory:adjust`
    43→- **Manufacturing**: Production order permissions
    44→- **Quality**: `iqc:inspect`, `iqc:approve`, `ncr:create`
    45→- **Finance**: `audit:read`, financial approval permissions
    46→- **Sales**: `customer:read`, `customer:create`

   221→## 6. Workflow Mapping
   222→
   223→### User Lifecycle Workflow (`UserLifecycle`)
   224→```
   225→INVITED → ACTIVATED → ACTIVE → LOCKED → DISABLED → ARCHIVED
   226→                ↑                   ↓
   227→                └───────────────────┘
   228→                    (unlock)
   229→```
   230→
   231→| Transition | Frontend Trigger | Backend Method | Audit Action |
   232→|---|---|---|---|
   233→| INVITED → ACTIVATED | User accepts invitation | `authService.acceptInvitation()` | `CREATE` |
   234→| ACTIVATED → ACTIVE | First login after activation | `authService.login()` (auto) | `LOGIN` |
   235→| ACTIVE → LOCKED | Admin clicks "Lock" | `userService.lockUser()` | `TRANSITION` |
   236→| LOCKED → ACTIVE | Admin clicks "Unlock" | `userService.unlockUser()` | `TRANSITION` |
   237→| ACTIVE → DISABLED | (Not implemented in frontend) | `userRepository.updateStatus()` | `TRANSITION` |
   238→| DISABLED → ACTIVE | (Not implemented in frontend) | `userRepository.updateStatus()` | `TRANSITION` |
   239→| Any → ARCHIVED | (Not implemented in frontend) | `userRepository.updateStatus()` | `TRANSITION` |
   240→| Auto-lock after 5 failed logins | (Backend auto) | `authService.login()` → `userRepository.lockUser()` | `ACCOUNT_LOCKED` |
   241→
   242→### Role Lifecycle
   243→Roles do not have a formal workflow. Status is managed via:
   244→- `roleService.create()` → status = `ACTIVE`
   245→- `roleService.update(id, { status })` → transition (not wired in frontend)
   246→- `roleService.delete()` → soft delete (`deleted_at` set)
   247→
   248→### Events Fired
   249→| Event | Trigger | Consumer |
   250→|---|---|---|
   251→| `UserRegistered` | User accepts invitation | Notification engine |
   252→| `UserLoggedIn` | Successful login | Security monitoring |
   253→| `UserLoggedOut` | Logout | Session cleanup |
   254→
   255→### Audit Trail
   256→- **Auth service**: 13 `auditService.log()` calls (login, logout, password change, invite, lock, unlock)
   257→- **User management service**: 17 `auditService.log()` calls (role assign, role revoke, user update, lock, unlock, assignment create/revoke, delegation create/revoke, preference set)
   258→- All audit entries include: `tenantId`, `correlationId`, `actorId`, `actorName`, `ipAddress`, `userAgent`, `action`, `entityType`, `entityId`, `before/after`

   288→### Business Rules (Backend-Enforced)
   289→1. **Email uniqueness**: `userRepository.findByEmail()` — checked during invite
   290→2. **Username uniqueness**: `userRepository.findByUsername()` — checked during registration
   291→3. **Password history**: Last 10 passwords cannot be reused (`PASSWORD_HISTORY_COUNT = 10`)
   292→4. **Password strength**: `z.string().min(12)` — Zod validation
   293→5. **Max failed logins**: 5 attempts → auto-lock (`MAX_FAILED_LOGINS = 5`)
   294→6. **Lock duration**: 30 minutes (`LOCK_DURATION_MIN = 30`)
   295→7. **Role name uniqueness**: `roleRepository.findByName()` — checked during create
   296→8. **System role protection**: `is_system` roles cannot be deleted (frontend checks)
   297→9. **Optimistic concurrency**: `version` field on roles, `If-Match` header on PATCH/DELETE
   298→10. **Tenant isolation**: All queries scoped by `tenant_id`

   336→## 9. Data Flow

   338→### User List Data Flow
   339→```
   340→User opens RBAC tab → 'users'
   341→    ↓
   342→useEffect triggers
   343→    ↓
   344→apiFetch('GET /api/v1/admin/users?page=1&search=...&status=...')
   345→    ↓
   346→Backend: authMiddleware → tenantMiddleware → requirePermission('auth:manage_users')
   347→    ↓
   348→Route: userManagementRoutes.get('/users')
   349→    ↓
   350→Service: userService.list({ page, search, status })
   351→    ↓
   352→Repository: userRepository (raw SQL SELECT FROM users WHERE tenant_id = $1 AND deleted_at IS NULL)
   353→    ↓
   354→Database: PostgreSQL (PGlite in dev)
   355→    ↓
   356→Response: { success: true, data: [...], meta: { total, page, pageSize } }
   357→    ↓
   358→Frontend: setUsers(json.data), setUsersTotal(json.meta.total)
   359→    ↓
   360→Render: User table with rows
   361→```
