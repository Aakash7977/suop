## 1. Business Purpose

The `Batch` entity represents a **manufacturing or procurement batch** — a quantity of homogeneous material produced or received together, identifiable by a unique batch number, with its own manufacturing date, expiry date, and quality status. Per Volume 0 Chapter 1 §2.4 (Challenges: "Which supplier supplied this batch?"), the batch is the **fundamental unit of traceability** in SUOP.

**Important distinction (per Volume 0 Chapter 4 Commitment #11)**:
- **Lot** = incoming material identified by supplier's lot number (procurement-side concept)
- **Batch** = produced material identified by internal batch number (manufacturing-side concept)

A finished-good **Batch** is composed of multiple raw-material **Lots** (via the `BatchIngredient` join table). This distinction is what makes the **5-minute batch recall KPI** (Ch 1 §2.8) achievable — forward/backward traversal alternates between Lot and Batch nodes in the traceability graph (per Ch 18 §18.5).

Every batch carries:
- **Origin** — manufacturing (internal) or procurement (supplier-supplied)
- **Genealogy** — parent batches (for semi-finished → finished) + ingredient lots (for manufactured)
- **Quality status** — passes/failed/on-hold/recalled
- **Lifecycle** — created → under QC → released → consumed/closed
- **Compliance** — FSSAI/HACCP traceability records (per Ch 18 §18.7)

A batch is the **indivisible unit** for:
- Recall operations (Ch 18 §18.6)
- FEFO picking (Ch 5 §5.16)
- Quality holds (Ch 5 §5.10)
- Expiry tracking (Ch 1 §2.4)
- Customer complaints investigation (Ch 18 enhancement)

## 3. Database Schema

| Attribute | Value |
|---|---|
| Schema | `transactional` (Batch is technically master data but treated as transactional due to lifecycle) |
| Table Name | `batches` |
| Prisma Model | `Batch` |
| File Location | `prisma/schema/transactional/inventory/batch.prisma` |
| Partitioning | None initially (medium volume — max ~500k batches over 7 years); future partitioning by `manufacturing_date` month if > 1M |
| Lifecycle | 8-stage Batch Lifecycle (per Ch 4 §4.7): CREATED → UNDER_QC → RELEASED → ON_HOLD → BLOCKED → RECALLED → CONSUMED → CLOSED |
| Immutability | Batch records are immutable after creation; status transitions create new audit entries, not edits |

## 7. Validation Rules

| # | Rule | Layer | Enforcement |
|---|---|---|---|
| 1 | `batch_number` must be unique per company | DB + App | Unique constraint |
| 2 | `barcode_value` must be globally unique | DB + App | Unique constraint |
| 3 | `manufacturing_date` required if `batch_type` IN (MANUFACTURED, REWORK) | DB | CHECK constraint |
| 4 | `receipt_date` required if `batch_type` IN (PROCURED, TRANSFER_IN, RETURN) | DB | CHECK constraint |
| 5 | `expiry_date` must be > `manufacturing_date` (if both set) | DB | CHECK constraint |
| 6 | `expiry_date` must be > `receipt_date` (if both set) | DB | CHECK constraint |
| 7 | `expiry_date` is auto-computed from `manufacturing_date + product.shelf_life_days` (if product has shelf_life_days) | App | Service-layer auto-computation |
| 8 | `supplier_id` required if `origin_type = EXTERNAL_PROCUREMENT` | DB | CHECK constraint |
| 9 | `purchase_order_id` and `grn_id` required if `origin_type = EXTERNAL_PROCUREMENT` | App | Service-layer validation |
| 10 | `production_order_id` required if `origin_type = INTERNAL_PRODUCTION` | App | Service-layer validation |
| 11 | `recipe_version_id` required if `origin_type = INTERNAL_PRODUCTION` | App | Service-layer validation (traceability) |
| 12 | `transfer_id` required if `origin_type = INTER_COMPANY_TRANSFER` | App | Service-layer validation |
| 13 | `return_id` required if `origin_type = CUSTOMER_RETURN` | App | Service-layer validation |
| 14 | `initial_quantity` > 0 | DB | CHECK constraint |
| 15 | `current_quantity` ≥ 0 (unless `product.negative_stock_allowed = true`) | App + DB | CHECK + service validation |
| 16 | `reserved_quantity + damaged_quantity + qc_hold_quantity` ≤ `current_quantity` | App | Service-layer validation (invariant) |
| 17 | `uom_id` must equal `product.base_uom_id` | App | Service-layer validation |
| 18 | If `requires_cold_chain = true`, `storage_temp_min_c` and `storage_temp_max_c` required | DB | CHECK constraint |
| 19 | `storage_temp_max_c` > `storage_temp_min_c` | DB | CHECK constraint |
| 20 | `qc_hold_reason` required if `qc_status` IN (FAILED, REJECTED) | DB | CHECK constraint |

## 15. Performance Notes

| Consideration | Guidance |
|---|---|
| Row count | Medium — max ~500k batches over 7 years; ~1000 new batches/week |
| Partitioning | None initially; partition by `manufacturing_date` month if > 1M rows |
| Cache strategy | Redis cache, TTL 5 min (shorter — quantities change frequently); barcode lookup cached separately with 1-hour TTL |
| Query patterns | By `company_id + batch_number` (lookup), `barcode_value` (scan), `product_id + expiry_date` (FEFO), `warehouse_id + status` (warehouse ops), `supplier_id + receipt_date` (supplier analysis) |
| Hot path | Batch lookup by barcode MUST be < 200ms (per Ch 20 §20.10) — dedicated index + Redis cache |
| FEFO query | Dedicated `idx_batches_fefo` index for fast FEFO picking |
| Recall query | `idx_batches_recall_query` + materialized path for < 60 sec recall (per Ch 18 Q108) |
| Genealogy | `genealogy_path` materialized for O(1) lookup; recursive CTE for full tree |
| Denormalized quantities | `current_quantity`, `reserved_quantity`, etc. updated by event handler on each ledger entry (per Ch 10 Q1 event-updated summary pattern) |
| `shelf_life_remaining_days` | Recomputed daily by scheduled job (Scheduler Service) — not real-time |
| N+1 prevention | Eager-load `product`, `supplier`, `warehouse` when listing batches |
