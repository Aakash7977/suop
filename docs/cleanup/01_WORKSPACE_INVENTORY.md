## 1. Purpose

This report provides a complete top-level inventory of the workspace, cataloguing every directory and significant file along with its size, purpose, retention status, and risk level. It serves as the foundational reference for all subsequent cleanup reports (02 through 10).

---

## 2. Methodology

The workspace was scanned at the top level using recursive directory traversal. Sizes are reported in human-readable form (MB / KB). Each entry has been classified into one of three retention categories:

- **KEEP** — Actively used, required for build, runtime, or version control.
- **NEEDS REVIEW** — Possibly stale, duplicated, or superseded. Requires human confirmation before deletion.
- **SAFE TO DELETE** — Confirmed obsolete; deletion will not affect the build, runtime, or history.

Risk levels indicate the impact *if the entry were removed in error*:
- **Low** — Removal has no operational impact.
- **Medium** — Removal may affect developer workflow or local history, but not production.
- **High** — Removal would break the build, runtime, or version control.

## 6. Observations

1. **Heavy footprint from dependencies.** `node_modules/` is the single largest directory and should never be cleaned manually; it is fully regenerated by `npm install`.
2. **Two parallel code trees.** `apps/` (current, 499 MB) coexists with `mini-services/` (legacy, 1.3 MB). The legacy tree has been superseded by `apps/backend/`.
3. **Multiple prisma schemas.** Three different `schema.prisma` files exist with different model counts. See Report 02 for full duplicate analysis.
4. **Old sprint artifacts.** The `download/`, `upload/`, `scripts/`, and `tool-results/` directories contain historical sprint exports and generator scripts that are no longer referenced.
5. **Design assets.** The `skills/` directory contains ~61 MB of HTML design templates that are not part of the ERP runtime but may be referenced by design tooling.
