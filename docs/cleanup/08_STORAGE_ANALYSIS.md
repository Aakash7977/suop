## 1. Purpose

This report provides a quantitative breakdown of workspace storage consumption. It identifies the largest contributors to the 1.8 GB footprint, separates recoverable from non-recoverable storage, and projects the post-cleanup workspace size. This analysis underpins the prioritization of cleanup actions in Report 09.

---

## 2. Methodology

Storage figures were obtained via recursive directory sizing. Each top-level directory was measured in isolation. Percentages are computed against the 1.8 GB total. Categories are assigned based on retention status (KEEP, NEEDS REVIEW, SAFE TO DELETE) per Report 01.

## 7. Git History Optimization (Bonus)

The `.git/` directory is 76 MB. After deleting large binaries (especially the 90 MB of test evidence images), running Git's garbage collection can shrink the pack files:

```bash
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git repack -ad
```

**Expected recovery:** 5–15 MB, depending on how much of the history is occupied by the now-deleted binaries.

> **Caution:** `git gc --aggressive` rewrites pack files and should be run when no one else is pushing to the repo. Coordinate with the team before running.

## 9. Key Findings

1. **93% of workspace storage is non-recoverable** (`node_modules/`, `.git/`, active source). This is expected and healthy.
2. **The single largest recoverable item is test evidence images** at ~90 MB — more than all other recoverable items combined.
3. **Conditional recovery (Phase 3) could yield an additional ~45 MB**, primarily from design templates, but requires human review.
4. **Git history optimization** can recover an additional 5–15 MB after cleanup.
5. **Without policy changes**, the workspace will continue to accumulate 10–15 MB per sprint, driven by test evidence and sprint archives.
